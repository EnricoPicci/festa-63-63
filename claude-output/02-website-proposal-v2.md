## Scelte tecniche / Technical Choices

### HTML/CSS/JS senza framework
Il sito sara costruito con **HTML, CSS e JavaScript puri** (vanilla), senza alcun framework. Motivazioni:
- Un singolo file HTML, un file CSS, un file JS: nessuna complessita nascosta
- Nessun build step, nessun `npm install`, nessun `node_modules`
- Chiunque puo aprire i file in un editor di testo e modificarli
- Deploy immediato: push su GitHub e il sito si aggiorna

### Perche NON usare Tailwind CSS
Tailwind e stato valutato e scartato per questo progetto. Le ragioni:

1. **Leggibilita dell'HTML compromessa**: Con Tailwind, un semplice elemento diventa:
   ```html
   <div class="bg-white rounded-lg shadow-md p-6 flex flex-col items-center space-y-4">
   ```
   Con CSS vanilla:
   ```html
   <div class="profile-card">
   ```
   Per quattro non-sviluppatori che devono trovare e modificare il testo, le classi utility di Tailwind sono un ostacolo.

2. **Build step non necessario**: Tailwind richiede un processo di compilazione (o un CDN da 300KB+). Per un singolo file HTML e una complicazione inutile.

3. **CSS moderno e sufficiente**: Le variabili CSS (`--color-prosecco: #d4a843`), flexbox, CSS grid, `clamp()` per la tipografia responsive, e le media query offrono tutto cio che serve per un sito mobile-first.

4. **Editabilita**: Con CSS vanilla, gli stili stanno in un file separato (`css/style.css`) che i quattro amici non devono mai toccare. Il contenuto resta pulito nell'HTML.

### Hosting: Netlify (consigliato) o GitHub Pages

**Netlify e la scelta raccomandata**, pur avendo il codice gia su GitHub. Ecco perche:

| Criterio | GitHub Pages | Netlify |
|----------|-------------|---------|
| Setup | 2-3 click nelle impostazioni del repo | 5-6 click (login con GitHub, seleziona repo) |
| Form RSVP (futuro) | **Non supportato** - serve un servizio esterno | **Integrato**: basta aggiungere `netlify` al tag `<form>` |
| Dashboard | Nessuna (tutto via GitHub) | Dashboard chiara con stato deploy e risposte ai form |
| Deploy preview | No | Si, per ogni pull request |
| Costo | Gratuito | Gratuito (100 GB/mese, 100 form/mese) |
| Drag & drop | No | Si (si puo deployare trascinando una cartella) |

**Il vantaggio chiave di Netlify e il supporto nativo ai form.** Quando vorremo aggiungere la pagina di conferma partecipazione (RSVP), bastera aggiungere al codice HTML:
```html
<form name="rsvp" netlify>
  <input type="text" name="nome" placeholder="Il tuo nome">
  <select name="partecipazione">
    <option>Ci saro!</option>
    <option>Non posso, ma brindo da lontano</option>
  </select>
  <button type="submit">Conferma</button>
</form>
```
Nessun backend, nessuna funzione serverless, nessuna API. Le risposte appariranno nella dashboard di Netlify.

**GitHub Pages resta un'opzione valida** se non si prevede di aggiungere il form RSVP. In quel caso e la scelta piu semplice possibile.

### Comportamento dinamico futuro (RSVP)
Come descritto sopra, con Netlify l'aggiunta di un form RSVP e banale. Se si usa GitHub Pages, le alternative sono:
- **Formspree** o **Tally**: servizi esterni gratuiti che gestiscono form senza backend
- **Google Forms** embedded: funzionale ma meno elegante

In ogni caso, il sito resta statico. Non serve riscriverlo.

### Test
Si, e possibile aggiungere test. L'approccio consigliato:

1. **Test della struttura HTML** con un semplice script che verifica:
   - Che tutte le sezioni richieste siano presenti
   - Che le immagini referenziate esistano
   - Che i testi in entrambe le lingue siano presenti
   - Che i link siano validi

2. **Strumento**: Uno script Node.js o Python (oppure un semplice script bash) che puo essere eseguito con un singolo comando, ad esempio:
   ```bash
   npm test
   ```
   oppure
   ```bash
   python test_site.py
   ```

3. **Test visivi** (opzionali): Si puo usare un tool come Playwright o Puppeteer per fare screenshot automatici su diverse risoluzioni e verificare che il sito sia responsive.

### Gestione immagini: facile come "drag and drop"
L'obiettivo e che aggiungere foto al sito sia semplice quanto metterle nella cartella `pictures/`. Per ottenere questo:

1. Il JavaScript del sito leggera automaticamente le immagini dalla cartella `pictures/` tramite una **lista configurabile** in un file separato (ad esempio `images.json` o direttamente in una sezione ben marcata dell'HTML)
2. Per aggiungere una foto bastera:
   - Mettere il file nella cartella `pictures/`
   - Aggiungere il nome del file alla lista (una riga)
   - Fare commit e push

**Nota**: Un sito puramente statico non puo "scannerizzare" una cartella. Serve una lista esplicita dei file. Ma questa lista sara semplicissima da aggiornare e ben documentata.

Un'alternativa e usare uno **script di build** leggero (ad esempio uno script bash o Node) che genera automaticamente la lista delle immagini dalla cartella. In questo caso si puo effettivamente solo droppare le immagini e lanciare lo script.

### Pagina singola HTML
Confermato: tutto in un singolo `index.html`. Il toggle IT/EN funzionera client-side, mostrando/nascondendo i testi nelle due lingue.

## Facilita di modifica per non-sviluppatori

Questo e un requisito chiave. Ecco come sara strutturato il codice per renderlo facile da modificare:

### Struttura del contenuto nell'HTML
Ogni sezione avra **commenti chiari** che indicano cosa modificare:

```html
<!-- ===================== -->
<!-- SEZIONE: I FANTASTICI QUATTRO / THE FAB FOUR -->
<!-- Per modificare i profili, cambia il testo qui sotto -->
<!-- ===================== -->

<section id="fab-four">
  <div class="profile-card">
    <img src="pictures/giovanni.jpg" alt="Giovanni">
    <h3>Giovanni Della Frattina</h3>
    <!-- ITALIANO -->
    <p lang="it">Il Filosofo con la Penna a Destra. Laureato in...</p>
    <!-- ENGLISH -->
    <p lang="en">The Philosopher with the Right-Wing Pen. Graduated in...</p>
  </div>
  ...
</section>
```

### Documentazione per la modifica dei contenuti
Verra creato un file `COME-MODIFICARE-IL-SITO.md` (e la versione inglese `HOW-TO-EDIT-THE-SITE.md`) nella root del progetto con istruzioni passo-passo:

1. Come trovare il testo da modificare (cercare i commenti-guida)
2. Come aggiungere una foto (mettere il file in `pictures/`, aggiungere alla lista)
3. Come fare commit e push (con screenshot per GitHub web interface)
4. Come verificare che il sito si sia aggiornato
5. **Regola d'oro**: solo i cambiamenti committati e pushati vengono pubblicati

### Flusso di lavoro sicuro
- Il sito si aggiorna **solo dopo un commit e push** su `main`
- Si puo usare la **GitHub web interface** per modificare i file direttamente dal browser, senza bisogno di strumenti di sviluppo locali
- Ogni modifica passa attraverso git, quindi e sempre possibile tornare indietro
